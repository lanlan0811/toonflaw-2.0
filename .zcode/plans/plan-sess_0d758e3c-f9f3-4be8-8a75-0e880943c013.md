# “基于分镜表”AI 短剧工作流审计结论与修复计划

## 一、审计结论

**该工作流正在实现，但目前只是“界面和部分后端已编码”，还不是可完整执行的工作流。**

当前源码已经包含：

- `storyboard` 项目类型、项目卡片标签和专用跳转。
- TXT、Markdown、DOCX 分镜表解析。
- 分镜表提交后自动创建内部占位剧本、原始角色/场景/道具、分镜和视频轨道。
- 原始资产提示词、原始资产图片、衍生资产图片、分镜图、视频提示词、视频等底层接口。
- `getProgress / prepareStepRequest / runStep` 工作流编排接口。

但用户现在无法执行整个流程，并不是单纯的模型配置问题，而是存在以下确定的断链。

### P0：当前运行环境直接阻断接口

1. **Electron 使用 `file://` 加载网页，但二开补丁直接执行 `fetch('/api/...')`。**
   - `scripts/main.ts:149-156` 使用 `loadFile()`。
   - `data/web/secondary-dev-patch.js:42-49` 没有复用主应用通过 `toonflow://getAppUrl` 获得的动态后端地址。
   - 结果是分镜解析、提交、列表和所有工作流请求都可能出现 `Failed to fetch`，而不是发往 Electron 启动的后端端口。

2. **当前提交的生产后端包没有包含新增路由。**
   - 源码路由在 `src/router.ts` 中已经注册。
   - `npm/yarn start` 实际执行 `data/serve/app.js`，当前该文件缺少 `/api/storyboardImport/*` 和 `/api/production/workflow/*`。
   - 结果是生产模式下相关请求直接 404。

### P0：业务链存在两个核心断点

3. **提交返回的内部 `scriptId` 被前端丢弃。**
   - `src/routes/storyboardImport/commit.ts:330` 已返回 `scriptId`。
   - `data/web/secondary-dev-patch.js:149-153` 只读取 `total`，没有保存 `scriptId`。
   - `prepareStepRequest.ts:178-181` 的衍生资产图明确要求 `scriptId`。
   - 多次导入产生多个内部剧本后，分镜图和视频也无法自动判断应处理哪个剧本。

4. **“生成衍生资产”按钮并没有创建衍生资产。**
   - UI 将 `polishDerivedAssetPrompts` 显示成“生成衍生资产”，见 `secondary-dev-patch.js:9`。
   - 后端该步骤只润色已经存在的衍生资产提示词，见 `prepareStepRequest.ts:150-155`。
   - 真正新增衍生资产的逻辑目前只存在于 Production Agent 的 `add_deriveAsset` 工具中，见 `src/agents/productionAgent/tools.ts:113-159`，没有接入 REST 工作流。
   - 因而项目没有衍生资产时，按钮只会返回“没有可执行对象”。

### P1：即使接口可达，状态和执行范围仍不可靠

5. 按剧本筛选资产错误地使用 `o_assets.scriptId IS NULL`，而实际关联存储在 `o_scriptAssets`，可能混入同项目其他内部剧本的资产。
6. `getProgress.ts:128-133` 用剧本数和原始资产数交叉比较，原始资产已存在时仍可能显示“可执行”。
7. 提示词润色没有独立进度，原始资产和衍生资产步骤状态会失真。
8. `shouldGenerateImage=0` 的分镜仍计入分镜图总数，导致步骤无法达到完成状态。
9. 视频按历史视频记录数与轨道数混合统计；重试后容易长期显示“部分完成”。
10. `prepareStepRequest` 大多只排除“生成中”，没有排除“已完成”，重复点击会重复生成图片和视频。
11. `batchGenerateImage.ts:119-127` 的分镜失败状态更新缺少 `await`，失败任务可能永久停留“生成中”。
12. 衍生资产图在文本模型失败时不会把图片状态改为失败；视频读取图片失败时也可能永久停留“生成中”。

## 二、当前真实业务语义

分镜表项目导入后的合理流程应是：

1. 解析和确认导入分镜表。
2. `commit` 自动创建原始资产、内部剧本、分镜面板和视频轨道。
3. 润色/确认原始资产提示词。
4. 生成原始资产图片。
5. 根据原始资产及分镜上下文真正创建衍生资产。
6. 润色/确认衍生资产提示词。
7. 生成衍生资产图片。
8. 基于分镜表及关联资产生成分镜图。
9. 生成视频提示词。
10. 生成视频。

因此当前页面上的“从分镜表生成原始资产”并不是一个真实提取动作：原始资产已经在提交阶段生成；该按钮实际只是润色提示词。页面文案和步骤需要按上述真实语义调整。

## 三、实施计划

### 阶段 1：恢复接口可达性

1. 修改 `data/web/secondary-dev-patch.js`：
   - 增加统一 API 地址解析函数。
   - Electron `file://` 环境通过 `toonflow://getAppUrl` 取得动态后端地址。
   - 浏览器同源环境继续使用相对 `/api` 地址。
   - 所有 parse、commit、list、workflow 请求统一走该请求函数。
2. 保留 Authorization，并使 `runStep` 内部请求兼容 header token 和 query token。
3. 使用 `scripts/build.ts` 从当前源码重新生成 `data/serve/app.js`，不手工编辑 bundle。
4. 构建后以路由 smoke test 确认 production bundle 中以下接口可访问：
   - `/api/storyboardImport/parse`
   - `/api/storyboardImport/commit`
   - `/api/storyboardImport/list`
   - `/api/production/workflow/getConfig`
   - `/api/production/workflow/getProgress`
   - `/api/production/workflow/runStep`

### 阶段 2：建立分镜表批次与 `scriptId` 上下文

1. `commit` 成功后保存返回的 `scriptId`，后续所有状态查询和执行都携带它。
2. 页面切换项目时清空解析结果、文件和旧项目的 `scriptId`，防止跨项目误提交。
3. 扩展分镜表列表接口，返回当前项目内部剧本/导入批次及其分镜数量。
4. 管理页增加“当前分镜表/导入批次”选择；已有多个内部剧本时不再模糊地跨剧本执行。
5. 后端校验项目存在且 `projectType=storyboard`；所有 `scriptId` 必须属于当前项目。
6. 资产范围统一通过 `o_scriptAssets` 关联查询，不再用 `o_assets.scriptId IS NULL` 推断剧本归属。

### 阶段 3：补齐真正的“生成衍生资产”步骤

1. 在工作流枚举中新增独立步骤 `generateDerivedAssets`，不再用 `polishDerivedAssetPrompts` 冒充。
2. 新增 REST 接口，复用并抽取 `add_deriveAsset` 的数据库逻辑：
   - 输入 `projectId + scriptId`。
   - 读取该分镜表批次的原始资产、分镜内容及资产关联。
   - 通过结构化 AI 输出建议的衍生角色/场景/道具。
   - 以“父资产 + 名称”去重，写入 `o_assets.assetsId` 和 `o_scriptAssets`。
   - 返回新增、更新、跳过数量及失败原因。
3. 保留后续独立步骤：
   - 润色衍生资产提示词。
   - 生成衍生资产图片。
4. 修正页面按钮顺序与文案：
   - 原始资产提示词
   - 原始资产图片
   - 生成衍生资产
   - 衍生资产提示词
   - 衍生资产图片
   - 分镜图
   - 视频提示词
   - 视频

### 阶段 4：修复工作流进度和可执行对象

1. `getProgress` 分开统计：
   - 原始资产是否存在。
   - 原始资产提示词状态。
   - 原始资产图片状态。
   - 衍生资产是否存在。
   - 衍生资产提示词状态。
   - 衍生资产图片状态。
2. 分镜图总数只统计 `shouldGenerateImage != 0` 的分镜。
3. 视频按“每条轨道当前有效结果”统计，不按所有历史 `o_video` 记录累加。
4. 默认执行只选择未开始和失败对象：
   - 已完成项不重复生成。
   - “重试失败项”和“强制重生成”使用明确参数。
5. 生成步骤增加真实前置条件：
   - 原始资产图要求提示词存在。
   - 衍生资产图要求衍生资产及提示词存在。
   - 分镜图要求关联资产状态可用；允许无参考图时明确提示。
   - 视频要求轨道提示词和已完成分镜图存在。
6. 前端根据状态禁用不可执行按钮，生成中时禁止重复提交。

### 阶段 5：修复生成任务的失败落库和数据完整性

1. 分镜图失败更新增加 `await`，确保写入 `生成失败 + reason`。
2. 衍生资产图把提示词生成和图片生成放在同一个完整 `try/catch` 中，任何阶段失败都更新图片状态。
3. 衍生资产图优先使用已确认的资产 prompt，不再无条件覆盖；只有 prompt 为空时才自动生成。
4. 视频生成将文件读取/base64 转换纳入错误处理，确保失败任务不会永久停留“生成中”。
5. 视频提示词组装加入分镜关联的原始/衍生资产上下文。
6. 校验视频模型需要的 resolution，避免直接把 `imageQuality` 的 `1K/2K/4K` 当作视频分辨率。
7. 分镜表导入时若传入 `src`，同步保存为 `filePath`，避免“已完成但没有图片文件”。

### 阶段 6：完善分镜表管理页

1. 原始资产和衍生资产分区展示：描述、提示词、图片、状态、错误原因。
2. 支持编辑资产描述和提示词。
3. 支持单项生成、批量生成、只重试失败项。
4. 解析预览补齐景别、镜头运动、台词、音效和备注。
5. 修复 TXT 中 `场景1/场景2/...` 场景美术参考解析。
6. 展示完整 warnings，不再因默认时长掩盖“缺少/异常时长”警告。
7. 去掉误导性的旧悬浮流程入口，保留项目卡片进入的全屏分镜表管理页。

## 四、验证方案

1. 执行 TypeScript 检查：`yarn lint`。
2. 执行生产构建：`yarn build`。
3. 对生成后的 `data/serve/app.js` 做路由可达性 smoke test。
4. 使用三个标准样例分别验证 TXT、Markdown、DOCX：
   - 字段解析。
   - 角色/场景/道具生成。
   - `scriptId` 返回与保存。
   - 分镜、轨道、资产关联落库。
5. 使用一个全新的 `storyboard` 项目验证完整闭环：
   - 导入分镜表。
   - 原始资产提示词。
   - 原始资产图。
   - 创建衍生资产。
   - 衍生资产提示词和图片。
   - 分镜图。
   - 视频提示词和视频。
6. 验证失败场景：无模型、模型调用失败、图片文件不存在、部分分镜不生成图片、重复点击、多次导入。
7. 回归小说项目和剧本项目，确认工作流筛选与状态统计未被分镜表改造影响。

## 五、预期结果

完成后，“基于分镜表”将不再只是一个能导入数据的注入页面，而会形成可执行且可恢复的闭环：

`分镜表导入 → 原始资产提示词 → 原始资产图 → 衍生资产 → 衍生资产提示词 → 衍生资产图 → 分镜图 → 视频提示词 → 视频`

每一步都有明确的当前分镜表批次、可执行对象、生成状态、失败原因和重试入口。
